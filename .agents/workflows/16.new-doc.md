📄 New Doc Workflow

Workflow này hướng dẫn từng bước tạo một file doc mới trong `docs/function-docs/` khi phát triển tính năng hoặc module mới.

**Chỉ dùng `Write` — vì đây là file mới hoàn toàn chưa tồn tại.**

---

## 📋 Quy trình 5 bước

---

### 1️⃣ Thu thập thông tin từ source code

Trước khi viết bất kỳ dòng doc nào, đọc kỹ source file:

```bash
# Xác định file nguồn
Read renderer/js/modules/[module-name].js
# hoặc
Read renderer/js/components/organisms/[component-name].js
```

**Checklist thu thập — đọc và ghi chú:**

| Hạng mục | Cần tìm trong code |
|---|---|
| **Mục đích module** | Comment đầu file, tên biến/function export |
| **State / Properties** | Các biến `let`/`const` ở scope module, object state |
| **Public functions** | Các function trong `return {}` hoặc `window.X = ` |
| **Lifecycle** | `init()`, `destroy()`, `bind()`, `unbind()` |
| **Event listeners** | `addEventListener`, `on()`, `document.addEventListener` |
| **Keyboard shortcuts** | `key === '...'`, `Mod+`, `ctrl+`, `meta+` |
| **IPC / Electron calls** | `window.electronAPI.*` |
| **localStorage keys** | `localStorage.getItem(...)`, `localStorage.setItem(...)` |
| **Dependencies** | Các module khác được gọi: `TabsModule.`, `AppState.`, v.v. |
| **Socket events** | `socket.on(...)`, `socket.emit(...)` |

> [!IMPORTANT]
> Đọc **toàn bộ** file source, không chỉ đọc một phần. Doc thiếu thông tin còn tệ hơn không có doc.

---

### 2️⃣ Xác định loại module → chọn template

| Loại module | Template cần dùng | Ví dụ tham khảo |
|---|---|---|
| **JS Module** (modules/) | Template A — Module | `TABS.md`, `EDITOR.md` |
| **Organism Component** (components/organisms/) | Template B — Component | `MARKDOWN_VIEWER.md` |
| **Service** (services/) | Template C — Service | `markdown_sync_handoff.md` |
| **Tính năng cross-module** | Template D — Feature | *(xem bên dưới)* |

---

### 4️⃣ Tạo file với `Write`

Sau khi soạn xong nội dung hoàn chỉnh trong đầu:

```
Write docs/function-docs/[MODULE_NAME].md
```

**Quy tắc đặt tên file:**
- Module JS: `TABS.md`, `EDITOR.md`, `SETTINGS.md` — tên module viết HOA
- Component Organism: `MARKDOWN_VIEWER.md`, `SIDEBAR_LEFT.md` — tên component viết HOA, dấu `_`
- Service: `MARKDOWN_SYNC.md` — tên ngắn gọn mô tả bài toán
- Feature cross-module: `FEATURE_[TEN].md` — prefix `FEATURE_`

> [!WARNING]
> Kiểm tra file **chưa tồn tại** trước khi `Write`:
> ```bash
> find docs/function-docs -name "[FILE_NAME].md"
> ```
> Nếu file đã tồn tại → dùng `/update-docs` thay vì `/new-doc`.

---

### 5️⃣ Đăng ký vào index

Sau khi tạo file, thêm entry vào `docs/function-docs/README.md`:

**Đọc file README trước:**
```
Read docs/function-docs/README.md
```

**Append dòng mới vào bảng Modules (dùng Edit, không Write):**
```
old_string: (dòng cuối cùng của bảng | ... | ... |)
new_string: (giữ nguyên dòng cũ + thêm dòng mới bên dưới)
| [NEW_MODULE.md](NEW_MODULE.md) | Mô tả ngắn gọn |
```

---

## ✅ Checklist trước khi hoàn thành

- [ ] Đã đọc toàn bộ source file
- [ ] Đã ghi chú đủ state, functions, shortcuts, localStorage keys
- [ ] Đã chọn đúng template (A/B/C/D)
- [ ] Tên file đúng convention (HOA, dấu `_`)
- [ ] Dùng `Write` (không phải `Edit`) vì file mới
- [ ] Timestamp `*Document — YYYY-MM-DD*` ở cuối
- [ ] Đã thêm vào `docs/function-docs/README.md` bằng `Edit`

---

## 🔗 Kết hợp với workflows khác

```
/new-doc [module]   → sau khi hoàn thành tính năng mới
/update-docs        → cho các thay đổi nhỏ trên module đã có doc
/changelog          → ghi lịch sử thay đổi code (song song, độc lập)
```

---

> [!TIP]
> Sử dụng `/new-doc` ngay sau khi hoàn thành tính năng, trong khi code còn fresh trong đầu — đừng để đến session sau mới viết doc.
>
> **Quy tắc vàng:** Doc tốt = AI đọc vào và biết ngay khi nào gọi function nào, không cần đọc source.
