🗣️ Discuss Workflow

**Reference:** `ARCHITECTURE.md` § Development Workflow

---

## 4️⃣ Steps

1. **Research** — Use read-only tools (grep, find, Read)
2. **Deep Dive** — Explain current logic, constraints, weak points
3. **Propose** — List specific files/changes + trade-offs + risks
4. **Wait** — Stop. Don't code until user approves `/implement` or `/smart-edit`

---

## ⚠️ Forbidden Tools

| Tool | Allowed? |
|------|----------|
| Read, grep, find | ✅ |
| Edit, Write, Bash (write) | ❌ |
| git commit, git push, rm, mv | ❌ |

**No code changes until user approval.** Even tiny fixes need approval first.

---

## 💡 Key Principles

- **Analysis-First** — Understand the problem deeply before proposing
- **No Changes Without Approval** — Not even comments or typos
- **Transparent Trade-offs** — Always list pros/cons of each option
- **Respect Architecture** — Follow 3-tier tokens, IIFE pattern, component hierarchy
