## 4️⃣ Steps

1. **Research** — Use read-only tools (grep, find, Read)
2. **Deep Dive** — Explain current logic, constraints, weak points
3. **Propose** — List specific files/changes + trade-offs + risks
4. **Wait** — Stop. Don't code until user approves `/implement` or `/smart-edit`

---

## 🔴 HARD STOP: Zero Code Execution Rule

**This workflow MUST NOT execute ANY code-modifying actions.**

### Allowed (Read-Only Only)
- ✅ `Read` — view file contents
- ✅ `grep`, `find`, `Bash` (queries only: `ls`, `git log`, `grep -n`)
- ✅ `Bash` (information gathering: `npm list`, version checks)

### Forbidden (ABSOLUTE BAN)
- ❌ `Edit` — ANY file modification
- ❌ `Write` — ANY file creation
- ❌ `Bash` (write operations): `rm`, `mv`, `mkdir`, `touch`, `echo >`, `sed`, `awk`
- ❌ `git commit`, `git push`, `git branch`, `git reset`
- ❌ `npm install`, `npm run` (build/test scripts)
- ❌ Any tool that modifies filesystem, git state, or project files

**Violation = Abort immediately and report to user.**

### Forbidden Actions (Even If Seems Helpful)
| Action | Why It's Forbidden | What to Do Instead |
|--------|-------------------|-------------------|
| "Fix typo I noticed" | No changes without approval | List typo in proposal, wait for `/implement` |
| "Run tests to verify" | Would execute code | Show test command user should run |
| "Try the fix locally" | Creates modified files | Explain what user should test manually |
| "Commit my analysis" | git state changes | Describe commit message, user commits it |
| "Update CHANGELOG" | File modification | Suggest changelog entry, user adds it |

---

## 📝 What Output is Allowed

**ALLOWED (Analysis only):**
- ✅ Text explanation of current code
- ✅ Code examples (in markdown blocks, showing what COULD change)
- ✅ Numbered lists of specific files/functions to modify
- ✅ Trade-offs table (pros/cons)
- ✅ Risk assessment
- ✅ Questions to clarify requirements

**FORBIDDEN (Execution artifacts):**
- ❌ "I've modified file X" (any file changes)
- ❌ "Running tests..." (execution output)
- ❌ "Linting shows..." (from running your own lint)
- ❌ "I committed..." (git operations)
- ❌ "Changes applied" (any success messages from write tools)

**If you see execution output from your tools, STOP immediately.**

---

## 💡 Key Principles

- **Analysis-First** — Understand the problem deeply before proposing
- **No Changes Without Approval** — Not even comments or typos
- **Transparent Trade-offs** — Always list pros/cons of each option
- **Respect Architecture** — Follow 3-tier tokens, IIFE pattern, component hierarchy

---

## 🛑 ABORT Conditions

If ANY of these happen, STOP immediately:

1. **Tempted to fix code** → Don't. List it in proposal instead.
2. **About to run npm/git commands** → Don't. Describe what user should run.
3. **About to use Edit/Write** → Don't. Call `/smart-edit` in proposal instead.
4. **About to modify files** → Don't. User will do it after approval.
5. **Analysis complete** → Stop. Output proposal. Wait for user direction.

**NEVER auto-proceed to implementation.** Always wait for explicit user command.

## ✅ Verification Checklist

Before finishing, verify:
- [ ] No `Edit` tool was called
- [ ] No `Write` tool was called
- [ ] No `Bash` write operations (`rm`, `mv`, `echo >`, `git commit`)

- [ ] No git state changed
- [ ] Output is 100% analysis text (no code execution)
- [ ] Proposal lists specific files/functions to change
- [ ] Trade-offs explained for each option
- [ ] User still needs to approve `/implement` or `/smart-edit`
