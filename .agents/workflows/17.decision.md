📌 Decision Workflow

Workflow này tạo một Decision Record — tài liệu ngắn gọn ghi lại **bối cảnh**, **lý do**, và **hệ quả** của một quyết định kỹ thuật hoặc thiết kế.

**Thư mục lưu:** `docs/decisions/`
**Format tên file:** `YYYYMMDD-[slug].md`

---

## Khi nào dùng `/decision`?

Ghi lại quyết định khi nó thuộc một trong các loại sau:

| Loại | Ví dụ |
|---|---|
| **Kiến trúc** | Chọn IIFE thay vì ES Module, không dùng framework UI |
| **Thiết kế API** | Function nhận path string thay vì object |
| **Chọn thư viện** | Dùng Mermaid v10 thay vì v9 |
| **Trade-off có chủ ý** | Hy sinh performance để đơn giản hóa code |
| **Từ chối giải pháp** | Không dùng Redux vì app đủ nhỏ |
| **Workaround có lý do** | Dùng deferred execution để defer vì DOM chưa ổn định |
| **Convention bắt buộc** | Mọi module phải export ra `window.*` |

**Không cần ghi lại:**
- Quyết định hiển nhiên (đặt tên biến, format code)
- Bug fix thông thường
- Thay đổi UI nhỏ không ảnh hưởng kiến trúc

## 📋 Quy trình tạo Decision Record

---

### 1️⃣ Xác định thông tin cần ghi

Trả lời 5 câu hỏi trước khi viết:

| Câu hỏi | Ghi chú |
|---|---|
| **Vấn đề gì cần giải quyết?** | Bối cảnh, constraint, áp lực tại thời điểm đó |
| **Các lựa chọn đã cân nhắc là gì?** | Ít nhất 2 options (kể cả "không làm gì") |
| **Chọn cái nào? Tại sao?** | Lý do chọn — điểm mạnh của option được chọn |
| **Đánh đổi gì?** | Điểm yếu / rủi ro của lựa chọn này |
| **Hệ quả là gì?** | Ảnh hưởng đến code, pattern, constraint tương lai |

---

### 2️⃣ Đặt tên file

**Format:** `YYYYMMDD-[slug].md`

**Quy tắc đặt slug:**
- Mô tả **vấn đề**, không mô tả giải pháp
- Ngắn (2-5 từ), dùng dấu `-`
- Dùng danh từ / cụm danh từ

```
✅ 20260426-scroll-sync-strategy.md
✅ 20260426-module-export-pattern.md
✅ 20260426-draft-storage-approach.md

❌ 20260426-dung-localstorage.md        (mô tả giải pháp)
❌ 20260426-quyet-dinh-scroll.md        (quá chung chung)
❌ 20260426-tai-sao-khong-dung-redux.md (quá dài)
```

---

### 3️⃣ Viết file theo template

Dùng `Write` để tạo file mới tại `docs/decisions/[tên-file].md`:

```markdown
# [Tiêu đề mô tả vấn đề — viết như một câu hỏi hoặc tên vấn đề]

**Date:** YYYY-MM-DD
**Status:** accepted
**Author:** [tên hoặc "session YYYY-MM-DD"]

---

## Bối cảnh

[Mô tả tình huống, áp lực, constraint dẫn đến cần ra quyết định này.
Viết như đang giải thích cho người mới — họ không biết gì về context lúc đó.
2-5 câu là đủ.]

---

## Các lựa chọn đã cân nhắc

### Option 1: [Tên option]
- **Ưu:** ...
- **Nhược:** ...

### Option 2: [Tên option]
- **Ưu:** ...
- **Nhược:** ...

### Option 3: [Tên option] *(nếu có)*
- **Ưu:** ...
- **Nhược:** ...

---

## Quyết định

**Chọn: Option X — [tên option]**

[1-3 câu giải thích lý do chính. Tập trung vào WHY, không phải WHAT.
Điều gì trong bối cảnh lúc đó khiến option này là lựa chọn đúng?]

---

## Hệ quả

**Tích cực:**
- [Lợi ích cụ thể]

**Tiêu cực / Trade-off:**
- [Rủi ro hoặc điều phải hy sinh]

**Constraint tương lai:**
- [Điều gì KHÔNG NÊN làm vì quyết định này — để tránh conflict]
```

---

### 4️⃣ Đăng ký vào index

Sau khi tạo file, **bắt buộc** cập nhật `docs/decisions/README.md`:

```bash
Read docs/decisions/README.md
```

Append dòng mới vào bảng Index bằng `Edit`:

```
old_string: "| *(chưa có entry)* | | | |"
new_string: "| YYYY-MM-DD | [slug.md](slug.md) | Mô tả vấn đề ngắn | accepted |"
```

Hoặc nếu đã có entry rồi, append sau dòng cuối cùng của bảng:

```
old_string: (dòng entry cuối cùng trong bảng)
new_string: (giữ nguyên dòng cũ + thêm dòng mới bên dưới)
| YYYY-MM-DD | [slug.md](slug.md) | Mô tả vấn đề ngắn | accepted |
```

## 🔄 Cập nhật Decision cũ

### Khi quyết định bị thay thế

**Bước 1 — Cập nhật file cũ** (dùng `Edit`, không `Write`):
```
old_string: "**Status:** accepted"
new_string: "**Status:** superseded by [YYYYMMDD-new-slug.md](YYYYMMDD-new-slug.md)"
```

**Bước 2 — Tạo file mới** với `/decision` như bình thường.
Trong phần "Bối cảnh" của file mới, ghi rõ đang thay thế quyết định nào và tại sao.

**Bước 3 — Cập nhật README** — sửa status dòng cũ + thêm dòng mới.

> [!WARNING]
> **Không bao giờ xóa file Decision cũ.** Lịch sử quyết định có giá trị — biết một approach đã được thử và thất bại tránh lặp lại sai lầm.

## 💡 Ví dụ Decision Record thực tế

**Tên file:** `20260426-module-export-pattern.md`

```markdown
# Module Export Pattern

**Date:** 2026-04-26
**Status:** accepted
**Author:** session 2026-04-26

---

## Bối cảnh

MDpreview chạy trên Electron với renderer process dùng vanilla JS không qua bundler.
Không có import/export ES Module nào hoạt động trực tiếp trong browser context này.
Cần một cách để các module giao tiếp với nhau mà không cần build step.

---

## Các lựa chọn đã cân nhắc

### Option 1: Global variables trần (window.x = ...)
- **Ưu:** Đơn giản, không cần pattern
- **Nhược:** Dễ bị ghi đè, không có scope isolation, khó debug

### Option 2: IIFE + window export
- **Ưu:** Scope isolation, chỉ expose API public, dễ đọc
- **Nhược:** Verbose hơn một chút

### Option 3: ES Modules với type="module"
- **Ưu:** Standard hiện đại
- **Nhược:** Không tương thích với Electron renderer context hiện tại, cần refactor lớn

---

## Quyết định

**Chọn: Option 2 — IIFE + window export**

App không dùng bundler và chạy trong Electron renderer context — ES Modules không khả thi
mà không có build pipeline. IIFE cho scope isolation đủ tốt với overhead tối thiểu.

---

## Hệ quả

**Tích cực:**
- Mọi module đều isolated, không leak biến ra global
- API public rõ ràng qua `window.ModuleName`

**Tiêu cực / Trade-off:**
- Mọi module phải wrap bằng IIFE — verbose
- Không có tree-shaking hay lazy loading

**Constraint tương lai:**
- Mọi module mới PHẢI dùng IIFE pattern và export qua `window.*`
- Không được dùng `import`/`export` keyword trong renderer JS
```
