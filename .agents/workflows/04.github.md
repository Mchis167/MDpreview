🚀 GitHub Release Automation Workflow

Workflow này tự động hóa quy trình release bằng cách đọc `CHANGELOG.md`, tổng hợp nội dung, tạo commit + push, và cập nhật phiên bản.

**Reference:** `ARCHITECTURE.md` § Development Workflow, `/changelog` workflow

---

## 📋 Quy trình thực hiện

### Bước 1: Kiểm tra Changelog (Validate Changelog)

Đọc `CHANGELOG.md` để xác minh:

```bash
grep -n "## \[Not Committed\]" CHANGELOG.md
```

**Kiểm tra:**
- ✅ Có section `## [Not Committed]`?
- ✅ Có các items trong `### Added`, `### Changed`, `### Fixed`?
- ✅ Định dạng đúng (gạch đầu dòng `-`)?

Nếu không có nội dung → **STOP**, hỏi người dùng.

### Bước 2: Tổng hợp Release Notes (Synthesize Release Notes)

Từ `CHANGELOG.md`, tạo release notes:

```markdown
## Release vX.Y.Z — 2026-04-25

### 🎉 Added
- Feature A
- Feature B

### 🔧 Changed
- Improved component X
- Refactored module Y

### 🐛 Fixed
- Fixed bug in Z
```

**Quy tắc:**
- Tóm tắt ngắn gọn (1-2 dòng per item)
- Giữ emojis từ CHANGELOG
- Xác định loại release: Major (v1.0.0), Minor (v1.1.0), Patch (v1.0.1)

### Bước 3: Xác định phiên bản mới (Determine Version)

```bash
# Đọc phiên bản gần nhất
grep "^## \[" CHANGELOG.md | head -1

# Ví dụ output: ## [1.5.0] — 2026-04-20
```

**Quy tắc versioning:**
- **Major** (v1.0.0 → v2.0.0): Thay đổi API hoặc kiến trúc lớn
- **Minor** (v1.0.0 → v1.1.0): Tính năng mới, không breaking
- **Patch** (v1.0.0 → v1.0.1): Bug fixes, improvements nhỏ

### Bước 4: Thực hiện Git Operations (Execute Git Commands)

**Chuẩn bị:**

```bash
# Kiểm tra git status — PHẢI là clean (hoặc chỉ có CHANGELOG changes)
git status

# Nếu có uncommitted changes → commit trước
git add .
git commit -m "chore: prepare release vX.Y.Z"
```

**Create commit:**

```bash
git add .
git commit -m "🚀 Release vX.Y.Z: [Feature A], [Bug fix B], [Improvement C]

Release notes:
- Added: Feature A, Feature B
- Changed: Refactored X, Improved Y
- Fixed: Bug Z

See CHANGELOG.md for details."
```

**Push to remote:**

```bash
git push origin main
```

**Verify:**

```bash
git log -1 --oneline
# Output: 1a2b3c4 🚀 Release vX.Y.Z: ...
```

### Bước 5: Cập nhật Changelog (Update CHANGELOG.md)

Sau push thành công, chỉnh sửa CHANGELOG.md:

**Thay thế:**
```markdown
## [Not Committed] — YYYY-MM-DD HH:mm
```

**Bằng:**
```markdown
## [vX.Y.Z] — 2026-04-25 14:30
```

**Ví dụ:**
```markdown
## [1.5.1] — 2026-04-25 14:30

### 🎉 Added
- Semantic radius support in tokens
- Module creation workflow

### 🔧 Changed
- Improved design system structure
- Updated Antigravity workflows
```

## 🛡️ Safety Checks (Bắt buộc)

| Kiểm tra | Trước Push | Sau Push |
|---------|----------|---------|
| `npm run lint` | ✅ Must pass | ✅ Verify |
| `git status` | ✅ Check clean | ✅ Verify clean |
| CHANGELOG format | ✅ Valid | ✅ Updated |
| Commit message | ✅ Clear | ✅ In history |

**Nếu bất kỳ kiểm tra nào fail → STOP, không push.**

---

## ⚠️ Nguyên tắc quan trọng

1. **Changelog-First Releases**
   - CHANGELOG.md là nguồn sự thật (source of truth)
   - Release notes được tạo từ CHANGELOG, không phải ngược lại

2. **Zero-Error Commits**
   - `npm run lint` **PHẢI** 0 errors trước khi commit
   - Không bao giờ commit nếu có linting issues

3. **Clear Versioning**
   - Semantic versioning: MAJOR.MINOR.PATCH
   - Major: API/architecture changes
   - Minor: New features (backwards compatible)
   - Patch: Bug fixes, internal improvements

4. **No Force Push**
   - Tuyệt đối không `git push --force`
   - Nếu cần undo → `git revert` hoặc tạo fix commit mới

5. **Transparent History**
   - Commit message phải clear và descriptive
   - Không bao giờ squash hoặc rewrite history

## 🧪 Verification Checklist

- [ ] CHANGELOG.md có `## [Not Committed]` section
- [ ] `npm run lint` passes (0 errors)
- [ ] Phiên bản mới được xác định đúng (Major/Minor/Patch)
- [ ] Commit message rõ ràng và bao gồm release notes tóm tắt
- [ ] `git push` thành công
- [ ] CHANGELOG.md được cập nhật với phiên bản mới
- [ ] GitHub releases page (nếu có) được cập nhật
